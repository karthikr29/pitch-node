### [x] 1. Production Logging & Error Tracking
**Severity: High — flying blind in production**

Every `catch` block in all 14 API routes either silently swallows the error or returns a generic 500 with no log. Only 1 `console.warn()` exists in the entire backend. When voice calls fail, sessions get deleted silently with zero trace of why.

**What to add:**
- Integrate **Sentry** (free tier is sufficient) — catches unhandled exceptions in both Next.js server and client
- Add `Sentry.captureException(error)` in API route catch blocks
- Set up a Sentry project, add `SENTRY_DSN` env var, run `npx @sentry/wizard@latest -i nextjs`

**Files involved:**
- `next.config.ts` (Sentry config wraps it)
- All files in `src/app/api/` (add Sentry capture to catch blocks)
- New: `sentry.client.config.ts`, `sentry.server.config.ts`, `sentry.edge.config.ts` (auto-generated by wizard)

**Verification:** Trigger a deliberate error in an API route, confirm it appears in the Sentry dashboard.

### 4. Environment Variable Validation at Startup
**Severity: Medium — missing vars fail silently mid-request, not at boot**

No startup validation exists. `server.ts` uses TypeScript non-null assertions (`!`) which do nothing at runtime. `PIPECAT_SERVICE_API_KEY` is interpolated directly into fetch headers with no null check — a missing var sends `Bearer undefined` silently.

**Required env vars to validate:**
```
NEXT_PUBLIC_SUPABASE_URL
NEXT_PUBLIC_SUPABASE_ANON_KEY
NEXT_PUBLIC_LIVEKIT_URL
PIPECAT_SERVICE_URL
PIPECAT_SERVICE_API_KEY
NEXT_PUBLIC_SITE_URL          (production only)
```

**What to add:**
- Install `@t3-oss/env-nextjs` (or write a manual Zod schema)
- Create `src/env.ts` that validates all vars at import time — throws at build/startup if any are missing
- Replace all `process.env.X` references in API routes with imports from `src/env.ts`

**Files involved:**
- New: `src/env.ts`
- `src/app/api/voice/create-room/route.ts`
- `src/app/api/voice/infer-role/route.ts`
- `src/lib/supabase/server.ts`
- `src/lib/supabase/client.ts`

**Verification:** Remove one env var from `.env.local`, run `npm run build` — should fail immediately with a clear error naming the missing var.

### 5. Unbounded Database Queries
**Severity: Medium — works at 100 rows, breaks at 10,000**

Three API routes return all rows with no limit:

| Route | Problem |
|---|---|
| `GET /api/personas` | Returns every persona — no `.limit()` |
| `GET /api/scenarios` | Returns every scenario — no `.limit()` |
| `GET /api/analytics/trends` | Returns every session up to 365 days — no `.limit()` |

**Fix:**
- `personas` route: add `.limit(100)`
- `scenarios` route: add `.limit(50)`
- `trends` route: add a hard cap of 500 sessions max regardless of the `days` param

**Files involved:**
- `src/app/api/personas/route.ts`
- `src/app/api/scenarios/route.ts`
- `src/app/api/analytics/trends/route.ts`

**Verification:** Confirm routes return data correctly after adding limits.

### 6. Health Check Endpoint
**Severity: Medium — silent failures, no uptime monitoring possible**

**Files to create:**

**Verification:** Hit `GET /api/health` in browser — should return 200 with JSON.

### 8. Password Reset Flow
**Severity: Low — users locked out if they forget password, not a security hole**

"Forgot password?" in the login page is disabled (`cursor-not-allowed`). No reset route exists. Users who sign up with email/password and forget it are permanently locked out. Google OAuth users are unaffected.

**What to build:**

**Step 1 — Forgot Password page** (`/forgot-password`):
- Email input form
- Calls `supabase.auth.resetPasswordForEmail(email, { redirectTo: .../auth/reset-password })`
- Shows "Check your email" success state after submit
- Supabase sends a time-limited link (1 hour expiry by default)

**Step 2 — Reset Password page** (`/auth/reset-password`):
- User lands here from the email link (Supabase injects session via URL hash)
- New password + confirm password fields
- Calls `supabase.auth.updateUser({ password: newPassword })`
- Redirects to dashboard on success

**Step 3 — Wire up the login page:**
- Change disabled span to `<Link href="/forgot-password">`

**Files to create:**
- `src/app/(auth)/forgot-password/page.tsx`
- `src/app/auth/reset-password/page.tsx`

**Files to modify:**
- `src/app/(auth)/login/page.tsx` — enable the link
- `src/app/(auth)/actions.ts` — add `resetPassword` and `updatePassword` server actions

**Verification:**
1. Click "Forgot password?" → lands on forgot-password form
2. Submit email → shows "Check your email" message
3. Click email link → lands on reset-password page
4. Enter new password → redirected to dashboard
5. Old password no longer works; new password does
